# 设计规范

## 1. 目标与范围

本规范用于约束 `Bookshop` 项目的日常开发。后续每次新增功能、修改代码、补充文档前，均需先对照本规范执行。

## 2. 技术栈约束（第一阶段）

- 框架：`Spring Boot 3.5.13`
- JDK：`17`
- 构建：`Maven 3.6.3`
- 持久层：`MyBatis`（后续引入 `MyBatis-Plus`）
- 数据库：`MySQL`（库名 `bookshop`，字符集 `utf8mb4`）
- 接口测试：`Postman`

## 3. 配置规范

- 默认使用 `application.yml`，不再使用 `application.properties` 作为主配置文件。
- 环境拆分：`application-dev.yml`、`application-test.yml`、`application-prod.yml`。
- 敏感信息（密码、密钥）不写入仓库，开发阶段允许本地临时值，后续改为环境变量注入。

## 4. 代码组织规范

- 根包：`com.bookshop`
- 分层目录：
  - `controller`：接口层
  - `service`：业务层
  - `mapper`：数据访问层
  - `entity`：实体对象
  - `dto`：接口入参对象（请求参数）
  - `vo`：接口出参对象（返回给前端或调用方）
  - `common`：通用能力层（统一返回体、异常、工具、常量等）
  - `config`：配置类
- 命名要求：类名大驼峰、包名全小写、表名小写下划线。
- 业务模块化要求：每一层下按业务域继续分子包（如 `book`、`user`、`login`），便于后续扩展和快速定位。

示例（推荐）：

```text
com.bookshop
  ├─ controller
  │   ├─ book
  │   ├─ user
  │   └─ login
  ├─ service
  │   ├─ book
  │   ├─ user
  │   └─ login
  ├─ mapper
  │   ├─ book
  │   ├─ user
  │   └─ login
  ├─ entity
  │   ├─ book
  │   └─ user
  ├─ dto
  │   ├─ book
  │   ├─ user
  │   └─ login
  ├─ vo
  │   ├─ book
  │   └─ user
  └─ common
      ├─ exception
      └─ enums
```

## 4.1 对标主流企业架构要求

- Controller 不直接操作数据库，必须通过 Service 调用 Mapper。
- DTO 与 VO 必须和 Entity 解耦，避免接口层直接暴露数据库对象。
- `common` 层优先提供统一返回结构，后续逐步补齐全局异常和错误码体系。
- 错误码采用“按业务域拆分枚举”方式（如 `common.enums.book.BookErrorCode`、`common.enums.user.UserErrorCode`）。
- 所有模块按“可扩展”设计，先满足最小闭环，再逐步增强。

## 4.2 注释与讲解要求

- 每个新增文件必须在类注释中说明“这个文件的作用”。
- 关键方法必须写方法注释，至少包含“输入/输出/行为”。
- 关键业务步骤加行内注释，解释“为什么这么做”，不写无意义注释。
- 命名应具备可读性，尽量做到“代码即文档”。

## 5. 开发流程约束（必须遵守）

1. 改动前先确认规范和目标。
2. 采用“小步提交”方式，一次只做一个可验证改动。
3. 每次改动后至少执行一项验证（启动/单测/接口测试）。
4. 发现问题先记录原因，再修复，不跳步堆功能。
5. 同步更新 `CursorChat` 下对应文档，保证笔记与代码一致。

## 6. 当前阶段实施策略

- 先打通“启动 -> 连库 -> 健康检查 -> 最小 CRUD”闭环。
- 暂不引入 Redis、MinIO、RabbitMQ，待第一阶段稳定后逐个接入。
