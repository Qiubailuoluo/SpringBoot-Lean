# 项目测试与架构概览（当前版本）

## 1. 项目定位

- 架构形态：`Spring Boot` 单体后端项目（可逐步演进）。
- 当前目标：优先保证认证链路稳定、可联调、可回归，并逐步补齐可观测与运维能力。

## 2. 当前架构分层

- 接口层：`controller`（按业务域分包：`book`、`user`、`login`、`audit`）。
- 业务层：`service`（认证、用户、图书、验证码策略、审计）。
- 持久层：`mapper + MyBatis`（MySQL）。
- 模型层：`entity / dto / vo` 解耦。
- 通用层：`common`（统一响应、trace、错误码）。
- 安全层：`Spring Security + JWT + Redis`。
- 异常层：`GlobalExceptionHandler + BusinessException`。

## 3. 主要技术栈

- 运行时：`JDK 17`、`Spring Boot 3.5.x`
- 安全认证：`Spring Security`、`JWT`、`BCrypt`
- 数据库：`MySQL 8`、`Flyway`
- 缓存：`Redis`（refresh token、黑名单、限流相关）
- 文档与联调：`Swagger3/OpenAPI`、`Postman`、`Newman`
- 日志：`Logback` + `MDC traceId`

## 4. 已实现模块能力（摘要）

- 登录认证：登录、刷新、登出、token 黑名单。
- 密码治理：修改密码、忘记密码重置、旧登录态失效。
- 验证码流程：发送策略（mock/stub）、限流、回执统一结构。
- 安全收口：`username + verifyTarget` 绑定校验。
- 审计查询：验证码发送回执查询接口。
- 基础业务：`Book/User` CRUD。

## 5. 当前测试情况

- 单元测试：已覆盖认证关键分支（登录、改密、重置、目标绑定校验）。
- 冒烟测试：有关键 API 鉴权冒烟用例（未登录访问、参数校验等）。
- 集合回归：`Login` 模块支持 `Postman + Newman` 自动执行。
- 注意点：验证码限流与密码状态会影响重复跑结果，建议先执行“基线恢复集合”再跑主集合。

## 6. 当前运维可见性现状

- 已具备：`traceId` 贯穿响应头与日志（基础排障可用）。
- 未具备：统一的浏览器 UI 观测平台（接口性能看板、用户操作日志看板）。
- 现状结论：具备“本地排障能力”，但尚未形成“运维平台化可视化能力”。

## 7. 下一步执行方向（按你的最新要求）

- 鉴权策略：当前阶段保持“登录可访问”模型，不强推角色/资源级授权。
- P0 重点改为：尽快建立可视化观测运维体系：
  - 链路追踪 UI：优先 `SkyWalking UI`（接口耗时、拓扑、调用链）。
  - 日志检索 UI：优先 `Kibana`（按用户、接口、traceId 检索日志）。
  - 用户操作日志报告：补充“登录/登出/改密/重置/关键业务操作”审计事件并可视化。

## 8. 推荐最小落地方案（先快后全）

1. 接入 `SkyWalking OAP + UI`（应用以 Java Agent 方式接入）。  
2. 采用“文件采集到 ES”的方式接日志（Filebeat/Fluent Bit -> Elasticsearch -> Kibana）。  
3. 统一日志字段：`timestamp`、`traceId`、`username`、`path`、`status`、`latencyMs`、`eventType`。  
4. 建立基础看板：
   - 接口性能：P95/P99、错误率、慢接口 TOPN。
   - 用户操作：按用户查询关键操作轨迹（登录、改密、重置、管理操作）。
