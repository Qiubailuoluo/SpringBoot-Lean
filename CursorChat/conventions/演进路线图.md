# 演进路线图（观测、安全、异步）

本文件描述与当前架构衔接的后续练习方向。**本阶段不引入新依赖、不改变现有接口行为**；与 `application.yml` + `application-{profile}.yml` 的叠加关系、以及 `TraceIdFilter` 的职责，见《设计规范》与《日志与观测规范》。

## 1. 与现状的关系

- **链路 ID**：`TraceIdFilter` 为每次请求生成 `traceId`，写入 MDC（键名 `traceId`，与 `TraceIdFilter.TRACE_ID_KEY` 一致）、请求属性及响应头 `X-Trace-Id`；失败时 `ApiResponse` 体也带 `traceId`。
- **安全**：已接入 Spring Security + JWT + Redis 最小闭环；`/health`、`/api/auth/login`、`/api/auth/refresh` 放行，其余接口默认需 Bearer token。
- **异步**：未使用 `@Async`；无独立线程池 Bean。

## 2. 当前里程碑与建议下一步

| 状态 | 内容 |
|------|------|
| 已完成 | 登录域分层骨架、`logback-spring.xml`、TraceId 与统一 `ApiResponse`、Book/User CRUD、约定文档与 Postman、Security + JWT + Redis 最小闭环 |
| **建议下一步** | **补齐 5.2 安全增强**：完善 refresh token 多端策略与风控规则，补齐改密/重置密码链路，并收敛认证审计日志 |
| 可并行/靠后 | §5.1 SkyWalking（观测增强）、§5.3 `@Async`（有明确异步需求再上）、§5.4 ES 日志平台化 |

验收建议：当前应满足 `POST /api/auth/login` 返回 `access + refresh token`，受保护接口在无 Token 时返回 401。下一阶段重点是“真实密码体系 + 会话治理”。

## 5.1 SkyWalking（进程级 APM）

**目标**：在 JVM 内接入 SkyWalking Agent，获得拓扑、指标与分布式追踪视图。

**依赖**：SkyWalking OAP + UI（或托管方案）；应用侧以 Java Agent 方式启动，不在 `pom.xml` 中堆业务代码依赖。

**与现有架构**：

- SkyWalking 的 TraceId 与日志 MDC 中的 `traceId` 来源不同；接入后应在《日志与观测规范》中约定：**排障时以何者为主、如何在日志中关联 SW Segment**（例如通过 SW 提供的 MDC 桥接或统一在文档中说明「日志关键字 + 时间窗口」对照 UI）。
- `TraceIdFilter` 与 SW Agent 可并存：前者保证业务统一响应体与轻量日志关联，后者负责全链路采样与可视化。

## 5.2 Spring Security + JWT

**目标**：保护业务接口，统一认证与鉴权；登录颁发 JWT，刷新/吊销策略按练习进度迭代。

**依赖**：`spring-boot-starter-security`、JWT 库（如 `jjwt` 或 `nimbus-jose-jwt`），可选持久化 refresh token（届时再增 `entity`/`mapper` 子包）。

**与现有架构**：

- **白名单**：需统一配置匿名路径，例如 `/health`、`/api/auth/login`（及静态资源等）；骨架阶段已采用 `/api/auth` 前缀，减少后续改 URL。
- **过滤器顺序**：引入 Security 后，`FilterChain` 顺序需明确：`TraceIdFilter` 与 `SecurityFilterChain` 的先后、以及异常是否仍由 `GlobalExceptionHandler` 统一包装，避免重复处理或吞掉 `AuthenticationException`。
- **当前实现**：已落地 JWT + Redis（access/refresh、黑名单登出、`/api/auth/refresh`）。
- **当前实现**：`user_account.password_hash` 已采用 BCrypt 校验。
- **待增强**：细化 refresh token 多端/设备策略，补齐改密/重置密码与认证审计日志。

## 5.3 `@Async` 与线程池

**目标**：将耗时或非关键路径操作异步化（如通知、审计日志），不阻塞 Web 线程。

**依赖**：`@EnableAsync` + 显式 `ThreadPoolTaskExecutor` Bean（或自定义 `Executor`）；禁止依赖默认单线程池在生产环境「凑合用」。

**与现有架构**：

- **异常**：`AsyncUncaughtExceptionHandler` 或包装 `Future`，避免异步任务静默失败。
- **MDC**：异步线程默认不继承 MDC；若日志需 `traceId`，需使用 `TaskDecorator` 传递 `traceId`（或 SLF4J MDC 的跨线程方案），与 `TraceIdFilter` 的约定保持一致。

## 5.4 Elasticsearch 日志平台化（规划）

**目标**：将应用日志从本地文件扩展到可检索、可聚合、可告警的平台，贴近生产排障方式。

**依赖**：Elasticsearch + Kibana（可选 Filebeat/Fluent Bit 或 Logstash 作为采集管道）。

**与现有架构**：

- 当前 `logback-spring.xml` 仍以控制台和滚动文件为主，这一层继续保留，作为基础审计和本地排障兜底。
- ES 阶段建议采用“文件采集入 ES”优先，减少应用层耦合；待需求明确再评估直写 ES appender。
- 日志字段建议统一：`timestamp`、`level`、`service`、`traceId`、`path`、`message`、`exception`，确保能按 `traceId` 与 SkyWalking 交叉检索。
- 需补充日志留存和索引生命周期（ILM）策略，避免存储无限增长。
