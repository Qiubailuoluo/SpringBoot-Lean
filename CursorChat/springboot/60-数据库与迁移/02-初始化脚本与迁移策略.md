# 初始化脚本与迁移策略

## 目标
说明当前项目如何管理初始化 SQL 和表结构变更。

## 脚本位置
- `resources/db/migration/V1__create_book_info.sql`
- `resources/db/migration/V2__create_user_account.sql`
- `resources/db/migration/V3__create_verification_dispatch_audit.sql`
- `resources/db/migration/V4__ensure_user_password_hash.sql`
- `resources/db/migration/V5__add_user_verify_target.sql`
- `resources/application.yml`（`spring.flyway.enabled: true`）

## 当前策略
- 以 Flyway 为主做版本化迁移，按 `V{版本}__说明.sql` 顺序执行。
- 保留 `resources/sql` 作为本地初始化/学习脚本，不作为生产迁移主路径。
- 迁移脚本需保证“可重复部署”和“失败可回滚”。

## 迁移执行流程图
阅读提示：从左到右看，先测试库演练与校验，再准备回滚，最后进入生产执行窗口。
```mermaid
flowchart LR
newSchemaNeed["NewSchemaChange(新的库结构变更需求)"] --> writeSql["WriteMigrationSql(编写迁移SQL)"]
writeSql --> testDbRun["RunInTestDb(测试库执行)"]
testDbRun --> verifyData["VerifyDataAndApi(验证数据与接口)"]
verifyData --> prepareRollback["PrepareRollbackSql(准备回滚SQL)"]
prepareRollback --> prodExecute["ExecuteInProdWindow(生产窗口执行)"]
prodExecute --> postCheck["PostDeployCheck(发布后检查)"]
```
## 图解摘要
- 结构变更先写脚本，再在测试库完整演练。
- 通过校验后必须先准备回滚脚本，再进入生产窗口执行。
- 生产执行后还要做发布后检查，形成闭环。

## 对应源码入口
- `resources/db/migration/V4__ensure_user_password_hash.sql`
- `resources/db/migration/V5__add_user_verify_target.sql`

## 执行建议
- 新迁移脚本命名带日期与目标：`yyyyMMdd-xxx.sql`。
- 上线前先在测试库演练，并保留回滚 SQL。
